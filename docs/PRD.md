# 原神圣遗物词条优化网站 — 产品需求文档（PRD）

## 项目信息

| 字段 | 值 |
|---|---|
| 项目名称 | `genshin_artifact_optimizer` |
| 语言 | 中文 |
| 技术栈 | Vite + React + MUI + Tailwind CSS |
| 原始需求 | 为《原神》玩家提供一个网站工具，用于深度优化角色圣遗物词条搭配，支持词条重分配优化与理想词条模板生成 |

---

## 一、产品目标

1. **词条优化精准化**：基于完整的原神伤害公式体系，在词条总数不变的约束下，为用户计算出数学最优的词条分配方案，并直观展示伤害提升幅度
2. **降低使用门槛**：支持从角色展柜自动导入圣遗物数据，让普通玩家无需手动录入即可获得优化建议
3. **全角色覆盖与可扩展**：支持当前全部原神角色，且新增角色仅需添加配置数据，无需修改代码逻辑

---

## 二、用户故事

1. **As a** 追求极限伤害的玩家，**I want to** 输入我当前圣遗物词条分布，**so that** 我能知道在词条总数不变的情况下如何重新分配词条以获得最高伤害提升
2. **As a** 正在刷圣遗物的玩家，**I want to** 输入角色和词条总数，**so that** 我能知道该角色的理想词条分配目标，指导我该保留哪些圣遗物
3. **As a** 拥有多套圣遗物方案的玩家，**I want to** 考虑队伍搭配对词条优化的影响，**so that** 我能在特定队伍配置下获得最合理的词条建议
4. **As a** 不想手动录入数据的玩家，**I want to** 通过输入UID自动导入角色展柜的圣遗物数据，**so that** 我能快速获得优化结果而无需逐条填写
5. **As a** 新角色上线后的玩家，**I want to** 网站快速支持新角色，**so that** 我能第一时间获得新角色的词条优化建议

---

## 三、需求池

### P0 — Must Have（核心功能，MVP 必须）

| 编号 | 需求 | 说明 |
|---|---|---|
| P0-01 | 完整伤害计算公式体系 | 实现原神完整伤害公式：基础伤害区、倍率区（含基于生命值/防御力）、增伤区、暴击区、抗性区、防御区、增幅反应区、剧变反应区等全部乘区 |
| P0-02 | 词条重分配优化 | 用户输入角色+当前词条分布，系统在词条总数不变约束下，计算最优词条分配并展示伤害提升百分比 |
| P0-03 | 理想词条模板 | 用户输入角色+词条总数，系统输出理论最优词条分配方案 |
| P0-04 | 手动录入圣遗物词条 | 用户可手动填写圣遗物的主词条、副词条类型及数值 |
| P0-05 | 角色基础数据 | 支持全部原神角色的基础属性、突破属性等数据 |
| P0-06 | 技能倍率输入 | 用户手动输入代表性技能的倍率数值，系统预留扩展接口 |
| P0-07 | 伤害提升展示 | 以百分比形式展示优化后的伤害提升幅度，与优化前对比 |

### P1 — Should Have（重要但非阻塞）

| 编号 | 需求 | 说明 |
|---|---|---|
| P1-01 | 角色展柜自动导入 | 对接 Enka Network 等第三方API，通过UID自动读取角色展柜中的圣遗物数据 |
| P1-02 | 队伍搭配支持 | 选择队友后，考虑队友提供的增益（如元素共鸣、角色技能增益等）对最优词条的影响；队友使用固定圣遗物模板 |
| P1-03 | 优化结果可视化 | 用直观图表（如雷达图/条形图）展示词条分配对比（当前 vs 优化 vs 理想） |
| P1-04 | 武器选择 | 选择角色装备的武器，纳入武器主属性和被动效果对伤害计算的影响 |

### P2 — Nice to Have（增强体验）

| 编号 | 需求 | 说明 |
|---|---|---|
| P2-01 | 圣遗物套装效果 | 支持选择圣遗物套装（如绝缘4、宗室4等），将套装效果纳入伤害计算 |
| P2-02 | 多技能对比优化 | 同时考虑多个技能（如普攻+大招）的加权伤害作为优化目标 |
| P2-03 | 数据本地缓存 | 将角色数据、优化结果缓存到 localStorage，减少重复输入 |
| P2-04 | 优化方案导出/分享 | 将优化结果生成图片或链接，方便分享至社区 |
| P2-05 | 命之座支持 | 考虑角色命座对技能倍率和机制的影响 |

### 4.2 页面1：词条重分配优化

**交互流程：**

1. **选择角色** → 下拉选择角色名（支持搜索过滤）
2. **输入技能倍率** → 填写代表性技能的倍率数值
3. **选择队伍**（可选）→ 添加1-3名队友，队友使用固定模板
4. **录入圣遗物词条** → 两种方式：
   - **自动导入**：输入UID → 选择角色 → 自动填充词条
   - **手动录入**：选择5件圣遗物，每件选择主词条 + 填写副词条
5. **点击"优化"** → 系统计算
6. **展示结果**：
   - 当前词条分配 vs 优化词条分配（数值对比）
   - 伤害提升百分比（大字突出显示）
   - 词条分配雷达图/条形图

### 4.4 交互细节

- **角色选择器**：支持拼音首字母和汉字搜索，显示元素图标
- **圣遗物录入**：副词条类型通过下拉选择（仅展示该部位可能出现的副词条），数值通过输入框填写
- **词条输入验证**：检查词条总数是否合理（0-45范围），副词条类型不能与主词条重复
- **优化计算**：前端本地计算，无需服务端；计算过程中显示加载状态
- **响应式设计**：支持桌面端为主，移动端可用

## 五、待确认问题

| 编号 | 问题 | 影响范围 | 建议 |
|---|---|---|---|
| Q1 | 优化算法选择：穷举搜索 vs 启发式算法 vs 梯度法？词条总数最多约45，每种副词条最多约6种有效类型，搜索空间约 C(45,6)≈8M | P0-02, P0-03 | 建议使用动态规划或枚举剪枝，45个词条分配6种副词条的搜索空间可控 |
| Q2 | 伤害公式中「基于生命值/防御力造成伤害」的角色（如那维莱特、千织）是否纳入P0？需要额外的公式分支 | P0-01 | 建议纳入P0，这类角色越来越多，是核心需求 |
| Q3 | Enka Network API 的调用频率限制和可用性如何保障？是否需要备用方案？ | P1-01 | 建议实现手动录入作为兜底，API不可用时提示用户切换 |
| Q4 | 队伍搭配中，队友的固定圣遗物模板数据从何而来？需要预置多少模板？ | P1-02 | 建议预置常见搭配模板（如双暴头+沙杯冠常规分配），后续可扩展 |
| Q5 | 是否需要考虑圣遗物副词条的有效值范围（如每条暴击率2.7%-3.9%）？实际优化时按固定值还是浮动值计算？ | P0-02 | 建议P0按固定值（每条取中间值）简化计算，P2可考虑精确值优化 |
| Q6 | 角色数据（基础属性、突破属性等）的维护策略？游戏版本更新时如何同步？ | P0-05 | 建议数据驱动设计，角色数据存为JSON配置文件，社区PR更新 |
| Q7 | 纯本地工具的条件下，是否需要考虑计算性能？部分角色（如涉及多种反应加成）的计算可能较复杂 | P0-01 | 建议使用Web Worker避免阻塞UI，对复杂计算场景做性能测试 |
